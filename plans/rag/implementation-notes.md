# RAG Pipeline — Implementation Notes

This document explains the implementation details for the local RAG (Retrieval-Augmented Generation) pipeline and vector database integration. It serves as the single point of reference for understanding the architecture and interactions of the RAG module.

## Architecture Overview

The RAG module bridges the SQLite metadata store and the Qdrant vector database. It reads `transcription.json` files generated by the transcription module, embeds their textual and production metadata into dense and sparse vectors, and upserts them into Qdrant for hybrid search.

## Files Created

### `backend/creator_joy/rag/__init__.py`
Package marker. Exports the public API: `RAGService`, `RAGSettings`, `search_segments`, and key data models (`SearchResult`, `SegmentResult`, `StructuralFilters`, `IndexRecord`, `IndexStatus`).

### `backend/creator_joy/rag/models.py`
Data models and settings.
- `RAGSettings(frozen dataclass)` — Configures storage root, database filename, Qdrant host/port, and model names. Uses `Qwen3` and `minicoil` models.
- `IndexStatus(StrEnum)` — Lifecycle states for indexing: `pending | processing | completed | failed`.
- `IndexRecord(frozen dataclass)` — Represents one row from the `rag_index` SQLite table.
- `StructuralFilters(dataclass)` — Used to pass exact match (keyword/boolean) and range filters into Qdrant. **Note:** Must be a mutable dataclass (not frozen) to allow partial instantiation.
- `SearchResult` and `SegmentResult` — DTOs returned by `search_segments()`, cleanly formatting Qdrant records for the LLM.

### `backend/creator_joy/rag/database.py`
SQLite ops for the `rag_index` table. Opens the **same `creator_joy.sqlite3`** database as Ingestion and Transcription.
Methods: `create_or_reset_index()`, `update_index_status()`, `get_index()`.
Keys off of `video_id` and tracks the ingestion progress and errors.

### `backend/creator_joy/rag/collection.py`
Contains `ensure_collection()`.
**Crucial Behavior:** Payload indexes *must* be created before any upserts happen. This file configures the Qdrant collection `video_segments` with:
- Two dense vectors (`dense_transcript`, `dense_production`) of dimension 4096.
- One sparse vector (`sparse_minicoil`) using IDF modifier.
- Payload indexes for all `StructuralFilters` fields.
- `is_tenant=True` exclusively on `project_id` for HNSW co-location efficiency.

### `backend/creator_joy/rag/embedder.py`
Encapsulates embedding logic using `sentence-transformers` and `fastembed`.
- `timecode_to_seconds()` — converts MM:SS to floats for range filtering.
- `build_production_description()` — flattens a segment's nested production metadata (shot types, camera angles, lighting, background) into a natural language sentence. This enables semantic search over visual styles (e.g., "cinematic shots").
- `DenseEmbedder` — Lazy-loads `Qwen/Qwen3-Embedding-0.6B`. Passes `prompt_name="query"` vs `"document"` depending on context.
- `SparseEmbedder` — Lazy-loads `Qdrant/minicoil-v1` using `SparseTextEmbedding` (from `fastembed`), NOT `TextEmbedding`. fastembed has two separate model registries: `TextEmbedding` covers dense models, `SparseTextEmbedding` covers sparse models. Using `TextEmbedding` for `minicoil-v1` raises a `ValueError` at runtime regardless of any `model_type` parameter. Returns fastembed `SparseEmbedding` objects with `.indices` and `.values`.

### `backend/creator_joy/rag/reranker.py`
Cross-encoder logic.
- `CrossEncoderReranker` — Lazy-loads `Qwen/Qwen3-Reranker-0.6B`. Sorts Qdrant's Prefetch results based on relevance to the query. Mutates candidate dicts to inject a `_rerank_score`.

### `backend/creator_joy/rag/ingestor.py`
Orchestrates reading the transcription JSON and pushing to Qdrant.
**Flow:**
1. Looks up the video in SQLite to grab `project_id`.
2. Reads `transcription.json` from the video's directory.
3. Builds `dense_transcript`, `dense_production`, and `sparse_minicoil` vectors for all segments.
4. Generates a deterministic `point_id` using a UUID5 namespace + `video_id:segment_id`. This makes re-indexing idempotent.
5. Constructs a flat payload (no nested dicts) so Qdrant payload filters can work effectively.
6. Calls `client.upsert()`.

### `backend/creator_joy/rag/retriever.py`
The unified search entrypoint `search_segments()` and its helpers.
**Three Modes:**
- **Mode 1 (Structural):** Triggered when `nl_query` is None. Uses `client.scroll()` (or `client.count()`). Translates `StructuralFilters` into Qdrant `FieldCondition`s. Must use `scroll_filter=` or `count_filter=`.
- **Mode 2 (Semantic):** Triggered when `filters` is None. Uses `client.query_points()` with `Prefetch` objects to query dense and sparse vectors simultaneously, fusing with Reciprocal Rank Fusion (RRF). Must use `query_filter=`.
- **Mode 3 (Hybrid):** Combines both. Passes the structural filter to the `Prefetch` objects to narrow down candidate points before applying vector similarity.

### `backend/creator_joy/rag/service.py`
Top-level orchestration: `RAGService`.
Wires together databases, the Qdrant client, embedders, and the retriever. Exposes `.index_video()` and `.search()`.
Updates the SQLite `rag_index` table with COMPLETED or FAILED statuses based on ingestion outcomes.

### `backend/creator_joy/skills/search_skill/skill.md`
Agent documentation. A system-prompt style document that explicitly instructs an LLM agent how to invoke `search_segments()`. Explains the three search modes, available operations (FETCH, COUNT, SUM_duration, GROUP_BY), and lists the entire field reference table with concrete examples to avoid common mistakes (e.g., using semantic search for a strict "shot type").

### `backend/tests/dev_test_rag.py`
End-to-end integration test.
Tests `index_video` followed by six search scenarios validating COUNT, FETCH, GROUP_BY, semantic search, and hybrid search.

## Call Graph

```text
dev_test_rag.py
  └─ RAGService()
        ├─ IngestionDatabase(db_path)
        ├─ RAGDatabase(db_path)
        ├─ QdrantClient()
        │    └─ ensure_collection()  [creates collection + payload indexes]
        ├─ DenseEmbedder(lazy load)
        ├─ SparseEmbedder(lazy load)
        ├─ CrossEncoderReranker(lazy load)
        └─ VideoIngestor(settings, db, embedders, qdrant)

  └─ RAGService.index_video(video_id)
        ├─ RAGDatabase.create_or_reset_index(PENDING -> PROCESSING)
        ├─ VideoIngestor.index_video(video_id)
        │    ├─ Reads transcription.json
        │    ├─ Embedders encode transcripts & production summaries
        │    ├─ Builds flat Qdrant payload & deterministic UUIDs
        │    └─ QdrantClient.upsert()
        └─ RAGDatabase.update_index_status(COMPLETED/FAILED)

  └─ RAGService.search(project_id, filters, nl_query, ...)
        └─ retriever.search_segments()
             ├─ _build_filter()
             ├─ Mode 1 -> _structural_search() -> QdrantClient.scroll() / .count()
             └─ Mode 2/3 -> _semantic_search()
                  ├─ Embed query
                  ├─ QdrantClient.query_points(prefetch=[sparse, dense_T, dense_P], fusion=RRF)
                  └─ CrossEncoderReranker.rerank(candidates)
```

## How to Run

1. **Start Qdrant:**
   ```bash
   docker run -p 6333:6333 qdrant/qdrant:v1.17.1
   ```
2. **Setup Environment:**
   ```bash
   cd backend
   source venv/bin/activate
   pip install -r requirements.txt       # Install newly added fastembed, sentence-transformers, qdrant-client
   
   cp .env.example .env                  # Ensure QDRANT_HOST=localhost is set
   export $(grep -v '^#' .env | xargs)
   ```
3. **Execute Test:**
   ```bash
   python -m tests.dev_test_rag
   ```
   **Expected Output:** Logs confirming Qdrant collection creation. 1. Status of indexing (COMPLETED). 2. Mode 1 FETCH results (MCU shots). 3. Mode 1 COUNT. 4. Mode 1 GROUP_BY dictionary. 5. Mode 2 Semantic search results. 6. Mode 3 Hybrid search results.

## Key Quirks & Implementation Details

- **Qdrant API Syntax:** We explicitly avoid `.search()` because it was removed in `qdrant-client==1.17.1`. The API requires `.scroll(scroll_filter=...)`, `.count(count_filter=...)`, and `.query_points(query_filter=...)`. Mixing up `filter=` vs `query_filter=` leads to silent failures, which is why the parameters are explicitly named.
- **Payload Flattening:** Qdrant handles payload indexing much better on flat dictionaries rather than nested structures. The `VideoIngestor` breaks down the `RichVideoTranscription` object into top-level keys like `shot_type` and `catch_light_in_eyes`.
- **Tenant Efficiency:** Only `project_id` has `is_tenant=True` in Qdrant's `KeywordIndexParams` to efficiently split the HNSW graph by tenant.
- **Idempotency:** A video can be re-indexed seamlessly because `make_point_id` hashes `video_id` and `segment_id` using `uuid5`, ensuring points overwrite their previous states instead of duplicating.
